# dos:implement-models

Generate dbt models, schema YAMLs, tests, and contract enforcement configuration for a data product. This skill reads the contract as its primary input and enriches from quality config, pipeline architecture, and scope document when available. It produces code organized by layer (staging → intermediate → marts) with testing and contract enforcement aligned to the target platform.

## Preamble

Before starting, establish context and validate inputs:

1. **Which data product?** Ask the user for the data product name (e.g., `orders`, `customer-360`). This determines the artifact path: `docs/data-products/<name>/`.

2. **Validate upstream artifact.** Run the validation script:

   ```bash
   python scripts/validate-upstream.py <product-name>
   ```

   If validation fails, report what's missing and suggest: "Run `/dos:define-contract` to create or complete the contract for this data product." Do not proceed with code generation until the contract passes validation. The script also reports availability of optional inputs (quality config, pipeline architecture, scope).

3. **Read the contract.** Extract schema: objects (tables), properties (columns), types, constraints, relationships.

4. **Read optional artifacts.** If available:
   - **Quality config** (`quality-config.md`) — extract dimensions, thresholds, and scoring for test generation.
   - **Pipeline architecture** (`pipeline-architecture.md`) — extract layering strategy and incremental configuration.
   - **Scope document** (`scope.md`) — extract modeling recommendation and consumption patterns.

5. **Check for existing code.** Search the project for existing dbt models for this data product. If models exist, read them, diff against the current contract and quality config, and propose updates to align. If no models exist, proceed with generation.

6. **Detect project layout.** Find `dbt_project.yml` to determine model paths, naming conventions, and target platform. Ask the user if the project structure is not detectable.

## Workflow

### Step 1: Detect Target Platform

Before generating any code, identify the target platform from `profiles.yml` or ask the user. Flag incompatibilities that affect code generation.

Refer to [cross-platform-sql.md](references/cross-platform-sql.md) for the full compatibility matrix and dispatch shim patterns.

**ClickHouse — flag before proceeding:**
- No merge incremental strategy — use `delete+insert` with lightweight deletes
- No Python models — SQL only
- CTEs fail with INSERT — ephemeral models will break in table model SELECT
- `ReplicatedMergeTree` deduplication prevents re-insertion after `delete+insert`
- Recommend: append-only patterns with materialized views for pre-aggregation

**Semi-structured data — flag before proceeding:**
If the contract schema includes JSON types, dbt's 38 cross-database macros have zero coverage for JSON path extraction, regex, and array flattening. Require dispatch shim implementations before generating models that depend on these operations.

### Step 2: Generate Staging Models

Generate one staging model per source table. Refer to [dbt-model-patterns.md](references/dbt-model-patterns.md) for layer conventions and naming patterns.

- **Naming:** `stg_<source>__<table>`
- **Materialization:** `view` or `ephemeral`
- **Content:** Rename columns, cast types, basic cleaning. No joins, no business logic.
- **Source reference:** `{{ source('<source_name>', '<table_name>') }}`

Use consistent `{{ source() }}` references that match the dbt source definitions generated by `dos:implement-source`.

### Step 3: Generate Intermediate Models

Generate intermediate models for complex business logic that is reused by multiple marts.

- **Naming:** `int_<entity>__<verb>` (e.g., `int_orders__enriched`)
- **Materialization:** `table` or `incremental`
- **Content:** Joins, enrichments, deduplication, pivots.
- **When to create:** Logic is reused by multiple marts, or transformation is complex enough to warrant isolation.

### Step 4: Generate Mart Models

Generate consumer-facing models matching the modeling recommendation from scope (or ask the user). Refer to [dbt-model-patterns.md](references/dbt-model-patterns.md) for modeling approach patterns.

- **Kimball (default):** `fct_<process>` for fact tables at atomic grain, `dim_<entity>` for denormalized dimensions with surrogate keys.
- **OBT/Wide table:** `<entity>_wide` — best as serving layer on Kimball core.
- **Entity-centric:** `<entity>` — domain-organized, one model per business entity.
- **Materialization:** `table` or `incremental`. Enable contract enforcement.

### Step 5: Generate Schema YAMLs

Generate schema YAML alongside each model. Refer to [dbt-testing-patterns.md](references/dbt-testing-patterns.md) for test mapping and severity configuration.

For each model:
- Column definitions with descriptions from contract
- `contract: { enforced: true }` on mart models
- Generic tests from quality config: `not_null`, `unique`, `accepted_values`, `relationships`
- dbt-expectations tests for distribution/range checks if quality config specifies them (use Metaplane fork)
- Severity and `store_failures` configuration based on quality dimension weights

### Step 6: Contract Enforcement

Refer to [contract-enforcement.md](references/contract-enforcement.md) for the false confidence warning and platform-specific enforcement behavior.

**Critical warning:** DuckDB enforces all constraints at build time, but Snowflake and Databricks treat most constraints as metadata-only. For every constraint the production warehouse does not enforce, add an explicit dbt data test. A contract passing locally can silently allow invalid data in production.

Apply the three-layer enforcement strategy:
- **CI-time:** `dbt build --select state:modified+` for breaking change detection
- **Build-time:** `contract: { enforced: true }` for schema validation preflight
- **Runtime:** dbt data tests for quality and constraint enforcement

### Step 7: Generate Unit Tests

Generate dbt unit tests (v1.8+) for critical transformation logic. Refer to [dbt-testing-patterns.md](references/dbt-testing-patterns.md) for unit test patterns.

Focus on: complex joins, conditional logic, edge cases (nulls, zero quantities, negative values, boundary conditions). Run in dev/CI only — static mock inputs waste production compute.

### Step 8: Update Specification Artifacts

After successful code generation, update upstream artifacts to reflect implementation state:

1. **Contract:** Set `status` to `implemented`, update `last_modified`, add changelog entry with generated file paths.
2. **Quality config** (if consumed): Update `status`, add changelog entry noting which quality dimensions have corresponding dbt tests.

### Step 9: Next Steps

End with a recommendation for the downstream skill:

1. **`/dos:review-pipeline`** — Audit the full pipeline (dlt extraction + dbt transformation) against best practices, SLA compliance, and observability coverage.

Present this option and explain what the review skill will assess.

## Iteration Bounds

If generated code fails validation or the user requests changes:

- **Cap at 3-5 iterations** per step. If valid output cannot be produced within this bound, surface the failure with diagnostics rather than retrying.
- **Loop detection:** If three consecutive iterations produce >90% similar output, stop and explain what's blocking progress. The problem likely exceeds this skill's capability.
